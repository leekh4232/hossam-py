# 배포방법 정리

이 프로젝트는 `pyproject.toml` 기반(빌드 백엔드: setuptools)으로 패키징됩니다.
(`setup.py` 는 사용하지 않습니다.)

## 수동 배포

```shell
rm -rf dist
python3 -m pip install --upgrade build twine
python3 -m build              # sdist + wheel 생성
python3 -m twine upload dist/*
```

## GitHub 자동 배포

커밋 메시지에 **`build`** 문자열이 포함된 채로 `main` 브랜치에 push하면,
아래 두 워크플로가 함께 실행됩니다.

| 목적 | 워크플로 파일 | 트리거 조건 |
| --- | --- | --- |
| PyPI 패키지 배포 | `.github/workflows/publish-pypi.yml` | `main` push + 커밋 메시지에 `build` 포함 (또는 수동 실행) |
| API 문서 배포(GitHub Pages) | `.github/workflows/mkdocs-gh-pages.yml` | `main` push + 커밋 메시지에 `build` 포함 (또는 수동 실행) |

> 커밋 메시지에 `build` 가 없는 일반 push에서는 두 워크플로 모두 **건너뜁니다.**
> GitHub Actions 화면의 **Run workflow** 버튼으로 언제든 수동 실행(`workflow_dispatch`)도 가능합니다.

설정 요구사항:

- PyPI 배포: 리포지토리 Secrets에 `PYPI_API_TOKEN`(PyPI API 토큰)이 등록되어 있어야 합니다.
- 문서 배포: 추가 Secret 없이 `GITHUB_TOKEN` 으로 `gh-pages` 브랜치에 배포합니다. (저장소 Pages 소스를 `gh-pages` 로 설정)

### 사용 방법

```bash
# 변경사항 커밋 후 main 브랜치로 푸시 (메시지에 build 포함)
git add -A
git commit -m "build: release 0.5.16"
git push origin main
```

### 워크플로 동작 개요

- **PyPI**: checkout → Python 설치 → **`pyproject.toml` 의 패치 버전 자동 증가**(예: `0.5.16` → `0.5.17`)
  → `python -m build` 로 sdist+wheel 빌드 → `pypa/gh-action-pypi-publish` 로 업로드 →
  증가된 버전을 `chore: bump version to X [skip ci]` 커밋으로 main에 다시 기록.
  - 즉, **수동으로 버전을 올릴 필요가 없습니다.** `build` 커밋마다 패치 버전이 1씩 자동 증가합니다.
  - 버전 증가 커밋에는 `[skip ci]` 가 포함되어 워크플로가 재실행되지 않습니다.
  - `skip-existing: true` 이므로 혹시 동일 버전이 이미 존재하면 업로드를 건너뜁니다.
- **문서**: checkout → Python 설치 → `README.md` 를 `docs/index.md` 로 복사 →
  `mkdocs build` 로 `mkdocs.yml` 을 참조해 정적 사이트 생성 →
  `peaceiris/actions-gh-pages` 로 `gh-pages` 브랜치에 배포(CNAME: py.hossam.kr).

### 트리거 조건 변경 (선택)

모든 브랜치의 push에서 실행하려면 각 워크플로의 트리거를 다음처럼 변경합니다.

```yaml
on:
  push:
    branches:
      - '**'  # 모든 브랜치
```
